# Components

Reusable AMAZE inventory lives here. Room `BUILD.md` files should reference
inventory IDs instead of inventing one-off sourcing notes.

## Files

| File | Purpose |
|---|---|
| `INVENTORY.md` | Common parts, supplier classes, price bands, durability, spares, and substitution notes. |
| `SOURCING.md` | How AMAZE chooses common suppliers, validates prices, and avoids fragile one-off parts. |
| `TECHNIQUES.md` | Reusable physical puzzle techniques: fit, route, align, sort, reveal, team flow, play profiles, frustration triggers, and promotion tests. |
| `BEAT-PACKAGES.md` | Reusable beat packages that connect techniques, devices, room data, evidence hooks, and promotion criteria. |
| `DEVICES.md` | Reusable build modules that implement techniques: rails, sockets, windows, overlays, pipe boards, final cabinets, and powered feedback. |
| `KITS.md` | Reusable prototype, operations, electronics, mechanical, transport, and maintenance bundles. |
| `MATERIALS.md` | Reusable craft and durable materials, construction recipes, child/teen maker fit, and upgrade paths. |
| `VISUALS.md` | Editable Excalidraw part maps and diagram standards for player face, hidden build, reset, failure, and admin recovery. |
| `DEVICE-REVIEW.md` | Review levels, risk bands, kill criteria, and bench-test script for devices. |
| `RELIABILITY.md` | Failure taxonomy, chaos probes, variability matrix, FMEA-lite, and bench result states. |

## Inventory rule

Every promoted room component should answer:

1. Can we buy or fabricate it from a repeatable source?
2. What is the current price band?
3. What is the spare plan?
4. What is the acceptable substitute if the preferred item is unavailable?
5. What fails first under public use and transport?

## Technique/device rule

Every promoted puzzle should name:

1. the **technique** players are learning or exploiting;
2. the **beat package** that turns that technique into reusable room evidence;
3. the **device** or assembly that implements it;
4. the inventory IDs for repeated parts;
5. the kit IDs that make it operable, testable, or transportable;
6. the materials and construction recipe for prototype and public-use versions;
7. the expected crowd profile and frustration trigger;
8. the likely reliability issues and chaos probes;
9. the staff bypass and reset check.
